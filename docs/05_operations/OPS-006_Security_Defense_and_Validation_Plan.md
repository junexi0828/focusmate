# OPS-006: Security Defense and Validation Plan

본 문서는 외부 공격(침입, 데이터 탈취, 정보 노출)에 대비하기 위한 **보안 통제 항목**과 **검증 방법**을 ISO 양식에 맞춰 정의한다. 운영 중 서비스에 영향을 최소화하는 것을 우선 기준으로 한다.

## 1. 목적 및 범위
- **목적**: 외부 공격에 대한 방어 능력과 정보 유출 방지 체계를 수립하고, 검증 가능한 기준을 명확히 한다.
- **범위**: API/웹, 인증/인가, 데이터베이스, 로그/모니터링, 배포/운영 환경.

## 2. 준수 기준 (ISO)
- **ISO/IEC 27001**: 정보보안 관리체계(ISMS) 요구사항.
- **ISO/IEC 27002**: 보안 통제 모범 사례.
- **ISO/IEC 27005**: 위험 관리 프로세스.

## 3. 위협 모델 요약 (외부 공격 관점)
- **데이터 탈취**: DB 덤프 유출, 백업 노출, 과도한 권한.
- **웹 정보 유출**: IDOR, 과도한 응답 데이터, 오류 메시지, 공개 엔드포인트.
- **주요 취약점**: SQLi, XSS, CSRF, SSRF, RCE, 파일 업로드, 경로 탐색.
- **계정 공격**: 크리덴셜 스터핑, 세션 탈취, 토큰 재사용.
- **운영 취약점**: 비밀키 노출, 로그에 민감 정보 기록, 미패치 의존성.

## 4. 보안 통제 요구사항 (핵심 체크리스트)
| 영역 | 통제 항목 | 기준 | 검증 방법 |
| --- | --- | --- | --- |
| 인증/인가 | 최소 권한, RBAC, 토큰 만료/갱신 정책 | 모든 엔드포인트에 권한 검사 | 인증/인가 테스트 |
| 입력 검증 | 서버측 스키마 검증, 파일 업로드 정책 | 화이트리스트 기반 검증 | SAST/DAST |
| 세션/토큰 | 탈취 방지, 재사용 탐지 | 재사용 시도 시 차단/로그 | 보안 시나리오 테스트 |
| 데이터 보호 | 암호화, 접근 통제 | DB 접근 최소화, 백업 보호 | 구성 점검 |
| 정보 노출 | 민감 데이터 마스킹, 오류 메시지 최소화 | PII 무노출 | 응답 점검 |
| 의존성 | 취약점 스캔 및 패치 | Critical/High 0건 | SCA 스캔 |
| 네트워크 | 방화벽/보안 그룹 최소 허용 | 포트 최소화 | 인프라 점검 |
| 로깅 | 보안 이벤트 기록 | 비정상 접근 경보 | 로그/알림 테스트 |

## 5. DB 탈취 방어 요구사항
1. **네트워크 분리**: DB는 내부 네트워크에만 노출.
2. **접근 최소화**: 애플리케이션 계정은 최소 권한만 부여.
3. **암호화**: 저장 데이터 및 백업 암호화.
4. **비밀키 관리**: 환경 변수/비밀 저장소 활용, 코드 내 하드코딩 금지.
5. **감사 로그**: 대량 조회/덤프 시도 탐지 및 알림.
6. **백업 보호**: 백업 파일 접근 제어 및 주기적 키 회전.

## 6. 웹 정보 유출 방어 요구사항
1. **응답 최소화**: 필요한 필드만 반환, 내부 식별자 숨김.
2. **IDOR 차단**: 리소스 소유권 검증을 모든 읽기/쓰기 요청에 적용.
3. **오류 메시지 통제**: 스택 트레이스/SQL 오류 노출 금지.
4. **Rate Limit**: 과도한 조회/수집 방지.
5. **CORS 정책**: 허용 오리진 최소화, 와일드카드 금지.
6. **보안 헤더**: CSP, HSTS, X-Content-Type-Options, X-Frame-Options.

## 7. 운영 중단 최소화 원칙
- **점진적 적용**: 보안 통제는 단계별 배포(릴리즈/피처 플래그).
- **읽기 전용 검증**: 운영 데이터 변경 없이 테스트 계정으로 검증.
- **부하 제한**: DAST/스캔은 트래픽 저부하 시간대 수행.
- **안전한 롤백**: 변경 전후 구성 백업 확보.

## 8. 검증/디버깅 계획
### 8.1 자동화 검증
- **SAST**: 코드 정적 분석으로 취약점 탐지.
- **SCA**: 의존성 취약점 스캔(npm audit, pip-audit 등).
- **DAST**: 테스트 환경에 대해 보안 스캐닝 수행.

### 8.2 시나리오 기반 테스트
- SQLi/XSS/CSRF/SSRF 시나리오 입력.
- 인증 우회 및 권한 상승 시도.
- 토큰 재사용/세션 고정 공격 시도.
- 대량 데이터 조회 및 속도 제한 검증.

### 8.3 결과 검증 기준
- **Critical/High 취약점 0건**.
- **민감 정보 노출 0건**.
- **보안 이벤트 로그 기록 정상**.

## 9. 모니터링 및 사고 대응
- **알림 기준**: 비정상 로그인, 권한 오류 급증, 대량 조회.
- **대응 절차**: 차단 → 로그 수집 → 영향 범위 분석 → 패치 → 재발 방지.
- **정기 점검**: 월 1회 보안 스캔, 분기별 권한 리뷰.

## 10. 완료 기준 (Done Definition)
1. 본 문서의 통제 항목이 코드/구성에 반영됨.
2. 자동화/시나리오 기반 검증 결과 기준 충족.
3. 운영 장애/성능 저하 없음.
4. 보안 로그 및 알림 체계 정상 동작.

## 11. 변경 관리
- 변경 시 **위험 평가(ISO 27005)** 수행.
- 변경 내역은 `docs/05_operations/OPS-005_Changelog.md`에 기록.

## 12. 적용 현황 (Security Hardening Status)
### 12.1 적용 완료
- **보안 헤더**: X-Content-Type-Options, X-Frame-Options, Referrer-Policy, Permissions-Policy 적용.
- **IDOR 차단**: 프로필 조회/수정/이미지 업로드는 본인 또는 관리자만 접근 가능.
- **쿠키 보안**: Refresh 토큰 쿠키는 프로덕션에서만 Secure 적용.
- **파일 업로드 보호**: 경로 정규화, 파일 크기 상한 적용.
- **로그 노출 방지**: 요청 로그에서 전체 URL 제거(쿼리 토큰 노출 방지).
- **정보 노출 최소화**: 유저 검색 결과에서 이메일은 관리자만 확인 가능.
- **WebSocket 보안**: 토큰 Query 대신 Subprotocol 사용, 서버측 사용자 실검증 적용.
- **Rate Limit 강화**: 민감 엔드포인트 별도 제한 추가.

### 12.2 검증 결과
- **보안 테스트 스크립트**: `scripts/testing/run_tests.sh --security` 실행 시도.
- **상태**: pytest 미설치로 실제 테스트 미수행 (환경 준비 필요).
