# 🚀 Roadmap: From NAS Prototype to Enterprise Production

이 문서는 현재의 "NAS 안정화 버전(Fallback Mode)"이 **왜 엔터프라이즈급이 아닌지**, 그리고 **코드를 갈아엎지 않고** 어떻게 엔터프라이즈급으로 확장하는지를 설명합니다.

## 1. 3줄 요약

1.  **코드 다시 짤 필요 없습니다.** 비즈니스 로직(`app/api`, `app/services`)은 그대로 유지됩니다.
2.  **드라이버만 교체하면 됩니다.** 현재 DB 충돌 문제는 `asyncpg`라는 특정 라이브러리와 `PgBouncer` 간의 호환성 문제일 뿐입니다. (`psycopg3`로 교체 시 해결)
3.  **NAS가 특수한 것입니다.** Redis 문제는 코드가 아닌 Synology NAS의 환경(CPU 스케줄링/커널) 문제입니다.

---

## 2. 현재 상황 vs 엔터프라이즈 목표

| 구분 | 현재 (NAS Stable) | 목표 (Enterprise/Production) | 변경 필요 사항 |
| :-- | :--- | :--- | :--- |
| **DB 연결** | **Session Mode (5432)** | **Transaction Mode (6543)** | 드라이버 교체 (`asyncpg` → `psycopg3`) |
| **동시 접속** | 최대 ~60명 | **수천/수만 명** (무제한) | 위와 동일 (Pooling 모드 활성화) |
| **실시간성** | 1분 주기 Polling | **즉시 반응 (Real-time)** | Redis 재활성화 (환경 이전) |
| **인프라** | Synology NAS | AWS / GCP / Docker | 인프라 이전 |

---

## 3. 핵심 오해와 진실

### Q1. "Transaction Mode 6543, Redis는 아예 사용 못하는 건가요?"
**A. 아니요, 사용할 수 있습니다.**
지금 못 쓰는 이유는 **"NAS 환경" + "현재 라이브러리 조합"** 때문입니다.
- **DB**: `asyncpg` 라이브러리가 Transaction Mode와 궁합이 안 맞습니다. 이를 업계 표준인 `psycopg3`로 바꾸면 6543 포트를 완벽하게 쓸 수 있습니다.
- **Redis**: NAS의 CPU/OS 특성 때문에 멈추는 것입니다. 리눅스 서버(AWS, Docker)에서는 쌩쌩 돌아갑니다.

### Q2. "우리 코드 전부 바꿔야 해요?"
**A. 절대 아닙니다.**
`SQLAlchemy`라는 ORM을 쓰고 있기 때문에, 코드의 99%는 DB 드라이버가 바뀌어도 그대로 작동합니다.
- **유지되는 것**: API 로직, 데이터 모델, 서비스 함수, 검증 로직 등 (99%)
- **바뀌는 것**: `requirements.txt` (라이브러리 변경), `session.py` (연결 설정 한 줄) (1%)

### Q3. "결국 몇 명밖에 접속 못하는 개인 프로젝트 아닌가요?"
**A. 지금 설정으로는 맞습니다. 하지만 확장은 쉽습니다.**
현재 Session Mode(5432)는 연결 하나를 한 명의 유저가 독점합니다. 그래서 60명 제한이 생깁니다.
하지만 **Transaction Mode(6543)**로 돌아가면, 연결 하나를 수백 명이 돌려씁니다(Multiplexing).
우리는 "안정화"를 위해 잠시 5432로 대피한 것이며, **드라이버 교체 작업(Phase 2)**을 통해 언제든 6543으로 복귀할 수 있습니다.

---

## 4. 확장 로드맵 (Steps to Enterprise)

### Phase 1: NAS 안정화 (현재 달성) 🏁
- **목표**: 24시간 죽지 않는 서버
- **전략**: 호환성 타는 기능 끄기 (Redis, Transaction Mode)
- **상태**: ✅ 완료 (하지만 성능 제한 있음)

### Phase 2: 드라이버 교체 (다음 단계, 코드 수정) 🛠️
- **목표**: Transaction Mode (6543) 재진입
- **작업**:
    1. `asyncpg` 제거 -> `psycopg3` (Async) 설치
    2. `DATABASE_URL` 스키마 변경 (`postgresql+asyncpg://` -> `postgresql+psycopg://`)
    3. 6543 포트 연결 테스트
- **결과**: **동시 접속자 수천 명 처리가 가능해짐**

### Phase 3: 인프라 독립 (프로덕션 배포) 🚀
- **목표**: 리얼타임 기능 복구
- **작업**:
    1. 백엔드를 NAS에서 분리 (AWS EC2, Docker Container 등)
    2. Redis 활성화
- **결과**: 채팅, 알림, 타이머 동기화가 실시간으로 작동

---

## 5. 결론

지금의 "기능 제한"은 **"현재의 위기(NAS에서의 502/400 에러)를 끄기 위한 전술적 후퇴"**였습니다.
코드는 건강하며, **라이브러리 교체(Driver Swap)** 하나만으로 엔터프라이즈급 확장성을 다시 확보할 수 있습니다.

**걱정하지 않으셔도 됩니다.**
지금 작성된 코드는 쓰레기통으로 가는 것이 아니라, 더 큰 확장을 위한 튼튼한 기반이 됩니다.
